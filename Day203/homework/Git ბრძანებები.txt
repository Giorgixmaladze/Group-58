4)ახალი ფაილები,რომლებსაც Git საერტოდ არ აკონტროლებს (untracked)

მაგ-"Untracked Files"-ეს ნიშნავს რომ, ეს ფაილები ახლახან არის შექმნილი და Git ჯერ ვერ აღიქვამს როგორც პროექტის ნაწილი

- git add -> არის ბრძანება, რომელიც ირჩევს/ამზადებს კონკრეტულ ცვლილებებს შემდეგი commit ისთვის


Git ში მუშაობისას ცვლილებების შენახვა (commit) პირდაპირ არხდება. მანამდე არსებობს შუალედური ეტაპი,რომელსაც Staging Area(მოსამზადებელი ზონა ) ჰქვია. git add ს სწორედ ამ ზონაში გადააქვს შეცვლილი ან ჩამატებული ფაილები



მოდი ამ პროცესის მარტივი მაგალითი მოვიყვანოთ:

ვთქვათ რომ ვგზავნით რაღაც ამანათს:

1. Working Directory (სამუშაო საქაღალდე): ეს ოთახია, სადაც ნივთები(ფაილები), ყრია. ზოგს ვცვლით ზოგს ახალს ვქმნით.

2.Staging Area (git add) : ეს არის ყუთი. ჩვენ ვირჩევთ რომელი ნივთი ჩავდოთ ყუთში გასაგზავნად.ყველაფრის ჩადება არაა სავალდებულო

3.Repository (git commit): ეს არის ყუთის დალუქვა და ფოსტაში ჩაბარება

- git diff -> არის თითქოს "იპოვე განსხვავება"(spot the difference) თამაში. მისი მთავარი დანიშნულებაა ზუსტად გვიჩვენოს, რა შეცვალე ფაილებში სიმბოლოების და ხაზების დონეზე

წინა მაგალითიდან, თუ git add არის ფაილების ყუთში ჩალაგება, git diff არის იმის შემოწმება თუ , რა დევს ამ ყუთში ან რა დაგვრჩა გარეთ.


-რას გვიჩვენებს git diff ის შედეგი?
    როდესაც ამ ბრძანებას გაუშვებ, ტერმინალში დაინახავ ტექსტს, რომელიც ასე იკითხება:

    -წითელი ფერი და  - ნიშანი ნიშნავს წაშლილ ხაზების
    -მწვანე ფერი და + ნიშანი ნისნავს დამატებულ ახალ ხაზებს.
    -თეთრი ტექსტი: კონტექსტი(უცვლელი ხაზები ცვლილების გარშემო),რომ მიხვდე სად მოხდა ცვლილება


 

-ძირითადი გამოყენების შედეგები:

    -git diff --> ადარებს სამუშაო საქაღალდეს Staging Areaს. გვიჩვენებს იმას, რაც შევცვალეთ, მაგრამ ჯერ არ დაგვიმატებია (git add)

    -git diff --staged --> ადარებს Staging Areaს და ბოლო commitს. გვიჩვენებს იმას რაც უკვე გაამზადე შესანახად

    -git diff HEAD --> ადარებს სამუშაო საქაღალდეს და ბოლო commit ს. გვიჩვენებს აბსოლუტურად ყველა ცვლილებას ბოლო შენახვის წერტილიდან.


    -git diff branch1 branch2 --> გვიჩვენებს სხვაობას ორ სხვადასხვა branch ს შორის.

-git commit 


*რა არის git commit ?
    წარმოვიდგინოთ რომ რთულ ვიდეო თამაშს ვთამაშობთ. სანამ "ბოსს" შევებრძოლებით, თამაშს ვინახავთ. თუ დამარცხდები,შეგიძლია ზუსტად იმ წერტილიდან დავბრუნდეთ სადაც შეინახე.


    -git commitიც მსგავსი რამეა
        ეს არის ჩვენი კოდის მდგომარეობის შენახვა.

        თითოეულ "შენახვას" აქვს თავისი სახელი(აღწერა), ავტორი და დრო.

        თუ რამე გააფუჭე, შეგიძლია ნებისმიერ დროს დაბრუნდე წინა commit ზე.



*როგორია მუშაობის პროცესი?

    -Working Directory(სამუშაო გარემო):აქ ჩვენ ვცვლით ფაილებს,ვწერთ კოდს.Git ხედავს, რომ რაღაც შეიცვალა, მაგრამ ეს ჯერ არაფერს არ ინახავს. 

    -Staging Area(მომზადების ზონა):ბრძანების git add . შენ ეუბნები Git-ს:"ეს ცვლილებები მომზადებულია შესანახად".

    -Commir(შენახვა): ბრძანებით git commit -m "აღწერა" ცვლილებები სამუდამოდ იწერება პროექტის ისტორიაში.



*რისგან შედგება თითოეული Commit?
    
    commit არ არის უბრალოდ ფაილების ასლი. ის შეიცავს მნიშვნელოვან მეტამონაცემებს

    -SHA-1 Hash: ეს არის უნიკალური ID, რომელიც Git მა მიანიჭა ამ კონკრეტულ მომენტს.მისი საშუალებით შეგიძლია იპოვო ნებისმიერი ძველი ვერსია.

    -ავტორი: ვინ გააკეთა ცვლილება(სახელი და იმეილი).

    -დრო: ზუსტად როდის მოხდა შენახვა.

    - Commit Message: მოკლე აღწერა, თუ რა შევცვალეთ.

    -Parent: მითითება წინა commit ზე, რაც ქმნის ისტორიის ჯაჭვს.



*რატომ არის ის მნიშვნელოვანი?

    -უკან დაბრუნების შესაძლებლობა : თუ ახალმა ფუნქციამ მთელი საიტი გააფუჭა, ერთი ბრძანებით ბრუნდები წინა მდგომარეობაზე

    -გუნდური მუშაობა: შენ ხედავ, ვინ რა შეცვალა და რატომ.



*git commit --amend 

    წარმოვიდგინოთ ასეთი რამ: გავგზავნეთ მნიშვნელოვანი წერილი კონვერტით, დავაწებეთ მარკა , ჩავაგდეთ საფოსტო ყუთში და... უცებ გაგახსენდა, რომ შიგნით ფოტოს ჩადება დაგვავიწყდა.
    ან წერილის თავში "გამარჯობა" ს ნაცვლად "გამარჯოაბ" გვიწერია.

    ჩვეულებრივ ცხოვრებაში მეორე წერილის გაგზავნა მოგიწევდა, მაგრამ git ში გვაქვს git commit --amend ეს არის "დროში დაბრუნების" და ბოლო გაგზავნილი წერილის ჩუმად ჩასწორების საშუალება.

    მოკლედ რომ ვთქვათ --amend გამოიყენება ყველაზე ბოლო commit ის შესაცვლელად




    -რაში გვჭირდება ის? (2 ძირითადი შემთხვევა)

    1. commit ის მესიჯში შეცდომა დავუშვით

        თუ commit გავაკეთეთ მესიჯით git commit -m "feat:login logic fixx", მაგრამ გვინდა რომ fixx ის ნაცვლად fix ეწეროს 

        უბრალოდ ჩავწერთ: git commit --amend -m "feat:login logic fix"

    2. ფაილის დამატება დაგვავიწყდა

        ეს ყველაზე ხშირი შემთხვევაა. commit გავაკეთეთ და უცებ ვხედავთ რომ ერთი ფაილი დაგვრჩა დასამატებელი, ნაცვლად იმისა რომ პატარა დეტალისთვის ახალი commit შექმნა , ვაკეთებთ შემდეგს:

        -უკან დაბრუნების შესაძლებლობა : თუ ახალმა ფუნქციამ მთელი საიტი გააფუჭა, ერთი ბრძანებით ბრუნდები წინა მდგომარეობაზე

        -გუნდური მუშაობა: შენ ხედავ, ვინ რა შეცვალა და რატომ.

        ასწორებ ფაილს ან ამატებ დავიწყებულს: git add .

        წერ ბრძანებას: git commit --amend --no-edit
        (სტეკი --no-edit ნიშნავს, რომ მესიჯის შეცვლა არ გინდა, უბრალოდ ფაილები დაამატე).

-git log


*რას აკეთებს git log?

    როდესაც ტერმინალში ვწერთ git log, Git გვიჩვენებს commitების სიას უკუქრონოლოგიური თანმიმდევრობით(უახლესიდან უძველესისკენ). თითოეული ცანაწერისთვის ის გამოიტანს შემდეგ ინფორმაციას:

    -Commit Hash(Sha-1): 40 სიმბოლოიანი უნიკალური კოდი. ეს არის commit ის "პირადი ნომერი"
    
    -Author: ვინ განახორციელა ცვლილება(სახელი და იმეილი)

    -Date:ზუსტი დრო და თარიღი

    -Commit Message:აღწერა, რომელიც შენ დაწერე git commit -m ბრძანების დროს



*რატომ გვჭირდება ის?
    -იმისთვის რომ გაიგო თუ რა ეტაპზეა პროექტი და რა დაემატა ბოლოს

    -ვნახოთ გუნდის რომელმა წევრმა შეცვალა კონკრეტული კოდი.

    -თუ გვინდა პროექტის ძველ ვერსიაზე დაბრუნება, ჯერ git log ით უნდა იპოვო იმ ვერსიის ჰეშ კოდი



*დამხმარეები

    -ხშირად git log ძალიან გრძელი და რთული წასაკითხია, ამიტომ დეველოპერები ვიყენებთ სხვადასხვა "დამხმარეებს"


    *თუ გვინდა, რომ თითოეული commit ი მხოლოდ ერთ ხაზზე დაეტიოს:git log --oneline


    *თუ გვაქვს ბევრი branch და გვინდა ვნახოთ , როგორ უერთდებიან ისინი ერთმანეთს გრაფიკულად: git log --oneline --graph --all


    *თუ მხოლოდ ბოლო 3 commit ის ნახვა გვინდა: git log -n 3


    *თუ გვაინტერესებს მხოლოდ ჩვენი ან კონკრეტული მეგობრის ნამუშევარი:git log --author="Giorgi"

git checkout:

- ეს ბრძანება გამოიყენება სამი ძირითადი მიზნისთვის:branchების შეცვლა, ფაილების აღდგენა და ისტორიაში ნავიგაცია


1.branch ების შეცვლა:

    ეს checkoutის ყველაზე გავრცელებული ფუნქციაა. როდესაც შენს პროექტში გავს რამდენიმე "პარალელური სამყარო" (მაგალითად, main და feature-login), ამ ბრძანებით გადადიხარ ერღი შტოდაბ მეორეზე.

    -ბრძანება: git checkout <branch name>

    -რას აკეთებს: Git ცვლის შენს ფაილებს ისე,რომ ისინი ზუსტად დაემთხვეს იმ branch ის მდგომარეობას,რომელზეც გადახვედი.





2. ახალი branch ის შექმნა და გადასვლა.


    ხშირად არის ისეთი შემთხვევები ხოლმე როცა გვინდა შევქმნათ ახალი branch და  თან მაშინვე მაგ branch ზე გადავიდეთ.

    -ბრძანება: git checkout -b <new branch name>

    -რას აკეთებს: შექმნის ახალ branch ს და ავტომატურად გადავყავართ მასზე. ეს ბევრად უფრი სწრაფია ვიდრე git branch და მერე git checkout ის გამოყენება.





3. კონკრეტული ფაილების აღდგენა:

    checkout მხოლოდ branch ების შესაცვლელად არაა. მისი საშუალებით შეგვიძლია აღვადგინოთ კონკრეტული ფაილები. მაგალითად თუ წავშალეთ რაღაც მნიშვნელოვანი კოდი app.js ში და გინდა დააბრუნო ისე,როგორც ბოლო commit ის დროს იყო:

    -ბრძანება: git checkout HEAD <file name>

    -რას აკეთებს: ის იღებს ფაილის იმ ვერსიას, რომელიც შენახულია ბოლო commit ში.
    

4. ისტორიის შემოწმება

    თუ გვინდა ვნახოთ ,როგორი იყო მთელი პროექტი სამი დღის წინ , შეგვიძლია გადავიდეთ პირდაპირ კონკრეტულ commit ზე

    -ბრძანება: git checkout <commit_hash>
